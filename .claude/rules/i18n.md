# i18n (next-intl, FR/EN, `localePrefix: as-needed`)

`src/i18n/routing.ts` declares locales `["fr", "en"]`, default `fr`. Because of `as-needed`, FR routes have no prefix (`/projects`) and EN routes are prefixed (`/en/projects`).

## Navigation imports

**Public components and pages MUST import from `@/i18n/navigation` instead of `next/link` / `next/navigation`** — this exports locale-aware `Link`, `useRouter`, `usePathname`, `getPathname`, `redirect`. Using the raw Next primitives breaks locale preservation across navigation.

Exceptions (use raw Next primitives): the admin tree (FR-only), `Matomo.tsx` (needs raw URLs for analytics).

## Pages under `[locale]/`

Pages destructure `params: Promise<{ locale }>`, then call `setRequestLocale(locale)` for static rendering. `generateMetadata` is async and uses `getTranslations({ locale, namespace })` from `next-intl/server`. Forgetting `setRequestLocale` forces the page into dynamic rendering.

## Translation message structure

Translation messages live in `src/messages/{fr,en}.json`. The convention is:

- Data-only libs (`src/lib/skills.ts`, `src/lib/timeline.ts`, `src/lib/uses.ts`) hold IDs / dates / positions only.
- Labels (titles, kind names, descriptions) come from translation keys keyed by the same ID, e.g. `Timeline.items.{id}.title`, `Skills.category.{id}`, `Uses.sections.{id}`.
- `getStats()` returns `{ key: StatKey, value, suffix }` — labels resolved via `t(\`stats.${key}\`)` in components.

## Validation messages

`src/lib/validations.ts` schemas use **error codes** (e.g. `"nameTooShort"`), not translated strings. Components map codes → labels via translation keys (`Contact.errors.nameTooShort`, `Comments.errors.pseudoTooShort`).
